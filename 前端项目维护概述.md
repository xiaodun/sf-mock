# 技术是最后的手段

由于前端负责产品的最终呈现，所以研发阶段的任何抖动都会受到波及。

- 产品会从需求本身演化出基本的解决方案，积极评估方案对代码的影响并反馈沟通，找寻一种满足业务需求且对代码破坏最小的解决方案，代码的好坏，一定程度决定了产品的迭代成本、使用体验。

- UI 会根据产品原型图定义设计稿，前端要基于已有实现衡量还原的复杂程度，更好的用户体验和 100%还原设计稿是两回事。

- 后端会定义接口标准，对于已有前后端交互规则的破坏是不能接受的，这不仅增加了前端的复杂度，一定程度反应了后端内部的混乱。

- 测试职责的定义是根据已有需求对产品的功能性进行把控，对于上述延申出的其它问题，需要确保测试按照正常的流程沟通，比如需求问题先和产品确认，UI 问题要先和设计师确认，然后才是前端做出调整。

# 文档的重要性

从哲学的角度来讲，项目维护时不可能停止的，因为时间是流动的，具体到一个团队上，需求在变化、技术在演变、人员在更替。

所以当一个新人进入到团队，第一次开发某个项目，如何快速了解已有的规则是什么很重要，否则他很有可能写出满足需求但导致系统混乱的代码。

- 项目在业务需求上的使用场景，这一定程度决定了技术选型
- 描述项目在业务中的节点位置，和其它项目之间的关系，如何产生关联。
- 项目的基本功能模块，日期格式化、本地缓存的处理等。
- 常见业务的运作流程，例如一个订单的创建等。
- 代码示例，如何搭建一个新页面、建立 ajax 链接等
- 项目的启动、开发、联调、测试、部署节点中的关键信息
- 遗留的历史问题，哪些问题由来已久，且出现问题的解决方式是固定的。
- 未执行的优化方案，发现项目中已存在的问题，已衡量的点、计划的解决方式等
- 关键的代码注释，流程节点注释、输入和产出注释、整体功能概述、使用场景描写等。

# 前端工程化

前端工程化要基于实现的统一，检验对规范的践行程度
尽可能减少人为的干预、流程的打断，让一切自然而然地进行。

- 统一技术框架、组件库，例如 UI、各种工具库。
- 独立与不同项目的通用代码库，避免实现的的重复性和多样性。
- 统一的代码格式化方式
- 统一提交格式，具体到模块-功能-改动原因
- 统一命名方式，使其具备良好的区分性。
- 建立基本编写规范， 列如公共模块用绝对路径，其余都是相对路径，如何组织后端接口等。
- mock 环境的搭建，依赖后端接口的成本是很高的
- 保证开发的流程一致性，减少实现成本，例如一些链接跳转到其它环境。
- 有时候最影响工作效率的不是技术本身，对工具的选择、了解程度也很重要

# 封装抽象的重要性

抽象是对共性的抽调，封装是对共性的聚合

如无必要，不要新增项目，如无必要，不要使用新技术，如无必要，不要使用新方案，合适的才是最好的，而不是新的、没用过的，练习不一定要在业务上。

如何整合现有的技术解决业务上的需求，很考验前端的能力，而封装本身最难的是度的把握，通常情况下，过度封装比不封装还要可怕

- 封装需具备通用性，是一类问题的解决
- 封装需具备扩展性，方便增添能力
- 封装需具备透明性，例如代码可定位，参数依赖明确。
- 封装需具备健壮性，例如对异常情况的处理，不引起超越错误本身的交互问题
- 对代码敏感，对于实现复杂、繁琐地方及时优化
- 积极的抽调公用组件、模块，一些为了代码分割而独立出来的组件也是必要的
- 熟练掌握 Vue 及其生态，在合适的场景使用合适的方式
- 对于重复的逻辑要及时修成，进行必要的封装
- 减少魔法字符，比如路径跳转，可以统一管理

# 后续的迭代

及时对偏差做出修正，解决项目中的主要矛盾，渐进式的完善编程体验。
用户是不看代码，优化要尽量避免对产品功能的影响。

- 开发前确认本次需求的实现方案
- 开发前确认哪些优化可以在本次迭代中完成，占用的时间、影响的代码范围等
- 代码 review,成员之间要对彼此的实现有大致的了解
- 根据测试、线上的反馈问题总结日后开发需要优化的点
- 迭代完成后对项目做复盘，罗列出技术上需要解决的问题。
- 完善项目文档
